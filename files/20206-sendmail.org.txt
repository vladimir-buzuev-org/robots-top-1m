    <div class="box-overview container"><h3 class="box-overview__title">Overview</h3><div class="box-overview__description list-in-article"><p>The Sentrion platform is specifically designed for large, complex environments, but&nbsp;we make a subset of that solution available as an open source offering. Sentrion is not for everyone, but if you are using open source for a large complex environment and need an enterprise platform that will enable your messaging roadmap for years to come (virtualization, consolidation, cloud migration, etc.),&nbsp;visit our&nbsp;product page to see&nbsp;if&nbsp;<a href="node/12146">Sendmail&nbsp;Sentrion</a>&nbsp;is&nbsp;right for you.</p>

                      <div class="block-text-cols__cols"><div class="block-text-cols__cols__item"><h3 class="block-text-cols__title">Current Sendmail Open Source Release</h3><div class="block-text-cols__body list-in-article"><p>Sendmail&nbsp;8.15.2&nbsp;is available from&nbsp;<a href="ftp://ftp.sendmail.org">ftp.sendmail.org</a>. The release has a&nbsp;<a href="ftp://ftp.sendmail.org/pub/sendmail/sendmail.8.15.2.tar.gz">gzipped</a>&nbsp;tar file and a&nbsp;<a href="ftp://ftp.sendmail.org/pub/sendmail/sendmail.8.15.2.tar.gz.sig">PGP signature</a>&nbsp;file. The compressed/gzipped tar files are&nbsp;<a href="http://www.sendmail.com/sm/open_source/security/pgp_keys/">signed</a>&nbsp;by the&nbsp;<a href="https://www.proofpoint.com/sites/default/files/sendmail2015.asc">2015 signing key</a>&nbsp;PGP signature file. See the Security and PGP Signing Keys section for more information about how releases are signed.&nbsp;</p>

<p><strong>Important:</strong>&nbsp;Before downloading, please review the <a href="https://www.proofpoint.com/sites/default/files/sendmail-license.pdf">sendmail licensing terms</a>.</p>
</div></div></div></div>  </div>

</div><div class="field-item odd"><div class="entity entity-paragraphs-item paragraphs-item-text-columns bg--light-tan" id="paragraph-item-10856">
  <div class="content">
    <div class="block-text-cols container">
                      <div class="block-text-cols__cols"><div class="block-text-cols__cols__item"><h3 class="block-text-cols__title">Security and PGP Signing Keys</h3><div class="block-text-cols__body list-in-article"><p>Security advisories are issued by The Computer Emergency Response Team&nbsp;<a href="http://www.cert.org/" target="_blank">CERT</a>. Sendmail server related security problems should be sent to:</p>

<pre>
<strong>sendmail-security-<em>YYYY</em>@support.sendmail.org</strong>
</pre>

<p>Replace YYYY with the current year, e.g., 2015. This address is only for reporting security problems in sendmail. When reporting security problems, please use PGP-the public key is available in the file PGPKEYS of the sendmail distribution.</p>

<p><strong>Please do not use this address to report problems that are not related to the security of the sendmail server.&nbsp;</strong>Questions about stopping spam, how to set up your own certificate authorities, etc. should be posted in&nbsp;<a href="news:comp.mail.sendmail">comp.mail.sendmail</a>, and Unix-related security in the&nbsp;<a href="news:comp.security.unix">comp.security.unix</a>&nbsp;newsgroup.</p>

                      <div class="block-text-cols__cols"><div class="block-text-cols__cols__item"><h3 class="block-text-cols__title">Contact Us</h3><div class="block-text-cols__body list-in-article"><p>These addresses are for contributing patches or reporting problems about V8 sendmail. The members of these lists do not have the resources to support vendor versions. Before sending to any of these addresses, please check the FAQ and the files README, sendmail/README (on this web-site as Compiling Sendmail) and cf/README (on this web-site as the Configuration README pages) to see if they are already answered; about half of the questions received can be answered in this way.</p>

<h4>Notes</h4>

<ul>
	<li>Do not send us mail in HTML format, use plain text only (even multipart/alternative with an HTML part will be classified as spam by some of our members and hence an answer will most likely be delayed).</li>
	<li>Do not use 8bit characters in the Subject: nor use some encoding, e.g., =?GB2312?B?, but only plain 7bit ASCII without any charset encoding.</li>
	<li>If you send e-mail to sendmail.org, the answer will most likely not come from a system in the sendmail.org domain. If you use some anti-spam techniques (e.g., challenge-response systems) or you block hosts that are connected via DSL then please do not expect an answer. More and more often replies are blocked which is very annoying. Do not block this address or make sure that your system accepts at least STARTTLS secured mail. A list of IPs is available in the sendmail.org SPF record.</li>
	<li>If you use a challenge/response system, make sure that it does NOT send us a challenge when we reply to your question or when you receive mail from the announce list. We will not reply to those annoying mails.</li>
	<li>If you are using the blacklist from spamlist.org then please do not send us e-mail, the reply of the sendmail maintainer will not reach you.</li>
	<li>Do not send us mail in proprietary formats.</li>
	<li>The mailservers for support.sendmail.org now (2004-11-24) use the following DNSBLs:
	<ul>
		<li>sbl-xbl.spamhaus.org</li>
		<li>dnsbl.sorbs.net</li>
		<li>list.dsbl.org</li>
		<li>bl.spamcop.net</li>
	</ul>
	</li>
	<li>The mailserver for support.sendmail.org performs strict RFC checks, for example, it does not accept mail if the domain part has an MX record that points to an IP address (instead of hostname as required).</li>
</ul>

<p>If you have a question about sendmail, then please post it to the Usenet group&nbsp;<a href="news:comp.mail.sendmail">comp.mail.sendmail</a>. This newsgroup is dedicated to sendmail. Please make sure you check the usual resources before posting and follow the netiquette.</p>

<p>E-mail addresses to contact sendmail.org are (do&nbsp;<em>not</em>&nbsp;send questions about sendmail to these addresses, see above instead; replace&nbsp;YYYY&nbsp;with the current year, e.g., 2006, in all of these addresses):</p>

<ul>
	<li><a href="mailto:sendmail-YYYY@support.sendmail.org">sendmail-YYYY@support.sendmail.org</a>&nbsp;for contributing patches, feature requests, and general comments but not questions how to use, install, or configure sendmail;</li>
	<li><a href="mailto:sendmail-bugs-YYYY@support.sendmail.org">sendmail-bugs-YYYY@support.sendmail.org</a>&nbsp;to report implementation bugs;</li>
	<li><a href="mailto:sendmail-faq-YYYY@support.sendmail.org">sendmail-faq-YYYY@support.sendmail.org</a>&nbsp;only for comments / questions about the FAQ. Please mark your mail clearly with "FAQ: item" where item is the entry in the faq to which you are referring. If it is a general comment about the FAQ, use "FAQ: general", if it is an addition, use "FAQ: new".</li>
	<li><a href="mailto:sendmail-security-YYYY@support.sendmail.org">sendmail-security-YYYY@support.sendmail.org</a>&nbsp;(use this only to report&nbsp;related bugs or problems in sendmail). Please do not use this to ask about problems with your configuration, including how to stop spam, how to set up your own certificate authority, how to make sendmail work with S/MIME, etc. All such questions should be asked in&nbsp;<a href="news:comp.mail.sendmail">comp.mail.sendmail</a>. Also, please do not tell us that you were able to forge mail by using telnet to connect to port 25; this is fundamental to the Internet design for SMTP, and not a sendmail bug. Please use encryption to send mail to this address.</li>
	<li><a href="mailto:sendmail-mirror-YYYY@support.sendmail.org">sendmail-mirror-YYYY@support.sendmail.org</a>&nbsp;for updates about a mirror (e.g., a new mirror or an address change).</li>
	<li><a href="mailto:webmaster-YYYY@support.sendmail.org">webmaster-YYYY@support.sendmail.org</a>&nbsp;for comments about the website.</li>

                      <div class="block-text-cols__cols"><div class="block-text-cols__cols__item"><h3 class="block-text-cols__title">DKIM</h3><div class="block-text-cols__body list-in-article"><p>The Domain Keys Identified Mail (DKIM) Internet standard enables email senders to digitally sign their messages so that receivers can verify that those messages have not been forged. The DKIM sender authentication scheme allows the recipient of a message to confirm a message originated with the senderâ€™s domain and that the message content has not been altered. A cryptography-based solution, DKIM provides businesses an industry-standard method for mitigating email fraud and protecting an organizationâ€™s brand and reputation at a relatively low implementation cost.</p>

<p>DKIM has been approved by the IETF as a draft standard (RFC 4871). The protocol was developed through the cooperation of Sendmail, Cisco Systems and Yahoo!</p>

<p>Since being approved by the IETF, a new open source project was started. The OpenDKIM Project is a community effort to develop and maintain a C library for producing DKIM-aware applications and an open source milter for providing DKIM service.</p>

<p>The project started from a code fork of version 2.8.3 of the open source&nbsp;<a href="http://sourceforge.net/projects/dkim-milter/" target="_new">dkim-milter</a>&nbsp;package developed and maintained by Sendmail, Inc.</p>

<p>The Sendmail&nbsp;<a href="node/12146">Sentrion Message Processing Engine</a>&nbsp;comes standard with OpenDKIM.&nbsp;More information is available at&nbsp;<a href="http://opendkim.org/" target="_new">opendkim.org</a>&nbsp;and&nbsp;<a href="http://dkim.org/" target="_new">dkim.org</a></p>
